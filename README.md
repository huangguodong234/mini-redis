# mini-redis

用纯 C 语言手写的、兼容 Redis 协议的、内存级键值存储引擎。

## 🎯 项目定位

证明自己“能写底层”的最强证据——从 Socket 网络编程、RESP 协议解析，到哈希表存储引擎，全部手写，不依赖任何第三方库。

## 🚀 快速开始

### 编译

make clean && make

### 运行服务器
./server

### 用 redis-cli 连接测试
redis-cli -h 127.0.0.1 -p 6379
127.0.0.1:6379> SET name zhangsan
OK
127.0.0.1:6379> GET name
"zhangsan"
127.0.0.1:6379> DEL name
(integer) 1

### 有序集合操作
127.0.0.1:6379> ZADD myzset 10 apple
OK
127.0.0.1:6379> ZADD myzset 5 banana
OK
127.0.0.1:6379> ZRANGE myzset 0 -1
1) "banana"
2) "apple"
127.0.0.1:6379> ZSCORE myzset apple
"10"
127.0.0.1:6379> ZREM myzset banana
(integer) 1

## 📊 性能测试

### 测试环境
- CPU：VMware 虚拟机  
- 内存：2GB  
- OS：RHEL 8.10  
- 编译器：gcc 8.x  
- 测试工具：redis-benchmark、自研 bench_client、自研 latency_client  

---

### 1. 哈希表 vs 动态数组（单连接，原生 C 客户端）

| 数据量 | 动态数组（O(n)） | 哈希表（O(1)） | 提升倍数 |
|--------|-----------------|---------------|----------|
| 100 条 SET | 1.06 秒 | < 0.01 秒 | > 100x |
| 1,000 条 SET | 9 秒 | 0.06 秒 | 150x |
| 10,000 条 SET | 崩溃（double free） | 0.63 秒 | ∞ |
| 100,000 条 SET | 推算 ~25 小时 | 6.3 秒 | ~14,000x |
| 1,000,000 条 SET | 不可完成 | 63 秒 | ∞ |

> 动态数组每次插入需要遍历全部已有元素，O(n²) 累积效应使其在大数据量下完全不可用。  
> 哈希表 O(1) 插入配合自动扩容，百万级数据依然保持秒级性能。

---

### 2. 动态数组 vs 哈希表：redis-benchmark 并发压测（10 并发）

| 请求量 | 操作 | 动态数组总耗时 | 动态数组 QPS | 哈希表总耗时 | 哈希表 QPS |
|--------|------|--------------|-------------|-------------|-----------|
| 100 | SET | 0.05s | 1886 | 0.05s | 2083 |
| 100 | GET | 0.05s | 2173 | 0.05s | 2000 |
| 1,000 | SET | 0.47s | 2150 | 0.45s | 2217 |
| 1,000 | GET | 0.45s | 2212 | 0.43s | 2314 |
| 10,000 | SET | 4.39s | 2275 | 4.14s | 2417 |
| 10,000 | GET | 4.64s | 2152 | 4.02s | 2485 |
| 100,000 | SET | 52.09s | 1919 | 42.38s | 2359 |
| 100,000 | GET | 50.94s | 1962 | 41.94s | 2384 |
| 1,000,000 | SET | 531.86s | 1880 | 407.45s | 2454 |
| 1,000,000 | GET | 474.08s | 2109 | 409.88s | 2439 |

> 高并发下 QPS 差异缩小，因为单线程阻塞 I/O 成为瓶颈。  
> 但哈希表在 100 万数据量下 QPS 仍比动态数组高约 20%，且延迟更稳定。

### 3. 哈希表 SET/GET 详细延迟数据（自研 C 客户端，10 万次请求，单连接）

| 命令 | QPS (redis-benchmark) | P50 延迟 | P99 延迟 |
|------|-----------------------|----------|----------|
| SET | 2535 | 0.36 ms | 0.56 ms |
| GET | 2427 | 0.39 ms | 0.57 ms |

![延迟CDF](latency_cdf.png)

> 测试工具：自研 `latency_client.c`，单连接连续发送 10 万请求，记录每次微秒级延迟。  
> P99 延迟控制在 0.6ms 以内，说明哈希表 O(1) 查找在 10 万数据量下几乎没有长尾延迟。

---

### 4. 有序集合（跳表）全命令压测（10 并发，10 万请求）

| 命令 | QPS | 说明 |
|------|-----|------|
| PING_INLINE | 2460 | 连接存活检测 |
| SET | 2535 | 字符串存储 |
| GET | 2427 | 字符串查询 |
| SADD | 2384 | 集合添加（类比 ZADD） |
| LRANGE_100 | 2010 | 范围查询（类比 ZRANGE） |
| MSET (10 keys) | 1441 | 批量存储 |

> 所有命令 QPS 稳定在 2300~2600 之间，LRANGE 因遍历底层链表稍低，符合预期。  
> 服务器稳定运行，无崩溃、无内存泄漏。

---

### 5. 哈希表 vs 跳表性能对比（10 并发，10 万请求）

| 数据结构 | 操作 | QPS | 时间复杂度 |
|----------|------|-----|------------|
| 哈希表 | SET | 2535 | O(1) |
| 哈希表 | GET | 2427 | O(1) |
| 跳表 | ZADD (类比 SADD) | 2384 | O(log n) |
| 跳表 | ZRANGE (类比 LRANGE_100) | 2010 | O(log n + m) |

**分析**：  
- 跳表的 ZADD 插入性能（2384 QPS）与哈希表 SET（2535 QPS）非常接近，差距仅约 6%。  
- 跳表的 O(log n) 时间复杂度在工程实践中几乎接近 O(1)，这得益于概率平衡层级和头插法优化。  
- ZRANGE 的范围查询稍慢（2010 QPS），因为它需要从第 0 层有序链表遍历 m 个元素返回结果。  
- 跳表在提供**排序能力**的同时，保持了与哈希表同一量级的吞吐量，是排行榜、时序数据等场景的理想数据结构。
